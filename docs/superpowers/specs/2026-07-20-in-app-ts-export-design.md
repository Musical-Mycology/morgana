# In-App TS Export — Design Spec

- **Date:** 2026-07-20
- **Status:** Design / spec (approved for planning)
- **Tier:** 1.5 — Hardening (see [`docs/2026-06-29-morgana-end-state-design.md`](../../2026-06-29-morgana-end-state-design.md) §14a, §16)
- **Issue:** [#10](https://github.com/Musical-Mycology/morgana/issues/10)
- **Author of record:** Chris Oltyan (brainstormed with Claude)
- **Scope:** surface the existing, already-tested `deckDocToModule` serializer through the editor UI. One new component, one toolbar pill, a small panel-state refactor. No changes to the serializer, the deck format, or the engine.

---

## 1. Problem & goal

The `deckDocToModule` bridge ([`lib/bridge/export-ts.ts`](../../../lib/bridge/export-ts.ts)) turns a
`DeckDoc` into an importable TS module (`export const scenes: Scene[] = […]`), but it is reachable
from **no UI** — it can only be called in code. This is the Tier-1.5 "in-app TS export" item
(end-state §14a / §16).

**Goal:** an **Export** control in the editor toolbar that opens a panel showing the generated TS
module for the current deck, with **Copy** and **Download** actions. Author a deck visually → get the
exact TS module a consumer (e.g. mm-website's `investor-hub/lib/deck/`) imports.

**Non-goals (unchanged from #10 / end-state):**
- **Import** (TS module / JSON → `DeckDoc`) and the full round-trip — Tier 2, separate spec.
- Emitting `meta`/`fonts` — `deckDocToModule` emits **`scenes` only**, by design (cross-repo concern).
- Option knobs (`constName`, `typesImport`) in the UI — ship defaults only (decided 2026-07-20).
- Any server/API route — the export runs **client-side** on the in-memory deck.

## 2. What the serializer already does (unchanged)

`deckDocToModule(doc, opts?)` is a pure function (no I/O, no server deps). With defaults it emits:

```ts
// Generated by Morgana from deck "<id>" — edit in Morgana, not here.
import type { Scene } from "@/lib/deck/types";

export const scenes: Scene[] = [ /* JSON.stringify(doc.scenes, null, 2) */ ];
```

`opts.constName` (default `"scenes"`) and `opts.typesImport` (default `"@/lib/deck/types"`) stay
available in code but are **not** surfaced in this UI. The default `typesImport` is the *consumer's*
path (mm-website), intentionally left as-is — the emitted module is for the consuming repo.

## 3. Architecture

Client-side only. The panel reads `doc` from the editor store ([`lib/editor/store.ts`](../../../lib/editor/store.ts))
and calls `deckDocToModule(doc)` in the browser — no route, no round-trip. Three pieces:

### 3.1 `components/editor/ExportPanel.tsx` (new)

Mirrors [`components/editor/DeckSettings.tsx`](../../../components/editor/DeckSettings.tsx) — same
`ed__inspector` container and null-guard idiom. Responsibilities:

- Read `doc` via `useEditor((s) => s.doc)`.
- The outer container is always `<div className="ed__inspector" data-testid="export-panel">` (both branches), so `export-panel` visibility is a stable selector.
- If `doc` is null → the container holds just `<p>No deck.</p>` (matches DeckSettings' guard) — no buttons.
- Otherwise compute `const module = useMemo(() => deckDocToModule(doc), [doc])`.
- Render inside the container: a title (`Export`), a **read-only** `<textarea data-testid="export-code">` containing `module`, and two buttons: **Copy** (`data-testid="export-copy"`) and **Download** (`data-testid="export-download"`).
- The textarea is read-only because the deck is the source of truth ("edit in Morgana, not here"). Any deck edit re-runs the `useMemo` and updates the panel live — no manual refresh.

### 3.2 Toolbar pill — `app/editor/page.tsx`

Add an **Export** pill (`className="ed__pill ed__pill--ghost" data-testid="export-toggle"`) next to the
existing "Deck settings" pill in `.ed__bar`.

### 3.3 Panel-state refactor — `app/editor/page.tsx`

Today the bottom-right slot toggles on `showSettings: boolean` (line 27, 61, 67):

```tsx
const [showSettings, setShowSettings] = useState(false);
// …
<button … data-testid="deck-settings-toggle" onClick={() => setShowSettings(v => !v)}>Deck settings</button>
// …
{showSettings ? <DeckSettings /> : <Inspector />}
```

Generalize to a single three-way state so the slot renders exactly one panel:

```tsx
type Panel = "inspector" | "settings" | "export";
const [panel, setPanel] = useState<Panel>("inspector");
const toggle = (p: Panel) => setPanel((cur) => (cur === p ? "inspector" : p));
// pills:
<button … data-testid="deck-settings-toggle" onClick={() => toggle("settings")}>Deck settings</button>
<button … data-testid="export-toggle"        onClick={() => toggle("export")}>Export</button>
// slot:
{panel === "settings" ? <DeckSettings /> : panel === "export" ? <ExportPanel /> : <Inspector />}
```

Clicking a pill opens its panel; clicking the same pill again returns to the Inspector. Panels are
mutually exclusive (only one shows), so opening Export hides the Inspector and Deck settings.

## 4. Actions & behavior

- **Copy** — `navigator.clipboard.writeText(module)`. On success the button shows a transient
  "Copied" label (~1.5 s via `setTimeout`) then reverts. On rejection (insecure context / permission
  denied) catch the error and fall back: programmatically select the textarea's text and show a
  "Copy failed — select + ⌘C" hint. Never throws to the user.
- **Download** — `new Blob([module], { type: "text/plain" })` → `URL.createObjectURL` → click a
  synthetic `<a download="<doc.meta.id>.ts">` → `URL.revokeObjectURL`. Filename is the deck id
  (e.g. `demo.ts`), matching the module's `from deck "<id>"` header.
- **Regeneration** — derived from `doc` via `useMemo`; no stale snapshot to refresh.
- **No-deck / empty** — the null-guard renders "No deck." and no buttons.

## 5. Testing

### 5.1 Unit — `tests/unit/export-panel.test.tsx` (vitest + jsdom + @testing-library/react)

Seed the zustand store via its `load` action, render `<ExportPanel />`, assert on the DOM:
- With a small 1-scene deck loaded → the `export-code` textarea's value contains
  `export const scenes: Scene[]` and the scene's data (e.g. the beat/action content).
- With no deck (`load` not called / doc null) → renders "No deck.", no `export-copy`/`export-download`.
- Copy button click calls a mocked `navigator.clipboard.writeText` with the module string.

(The serializer's own output shape is already covered by the existing `export-ts` unit test; these
tests cover the **panel wiring**, not the serializer.)

### 5.2 E2E — `e2e/export-ts.spec.ts` (Playwright, `default` project on `:3000`)

Follows the throwaway-deck pattern (`POST`+`PUT` create → `/editor?deck=<id>` → assert → `DELETE`),
matching the recently-converted specs. Runs as a normal parallel-safe spec — no special isolation.

- Create deck `e2e-export`, open `/editor?deck=e2e-export`.
- Click the **Export** pill → assert `export-panel` visible and `export-code` contains
  `export const scenes` plus the deck's content.
- Panel-exclusivity: with Export open, the Inspector is not visible; clicking **Deck settings**
  swaps to `deck-settings` (proves the 3-way state).
- **Download**: `page.waitForEvent('download')` around the Download click; assert
  `download.suggestedFilename() === "e2e-export.ts"`.
- Clipboard assertions are omitted from e2e (awkward/permission-gated cross-browser) — the copy
  *call* is covered by the unit test; the e2e asserts the button exists and the code content.
- `DELETE` the deck at the end.

## 7. Alternatives considered

| Alternative | Why not |
| --- | --- |
| **Modal / overlay** for the export | More room for long output, but introduces a modal pattern (focus-trap, escape, backdrop) the codebase doesn't have yet — new surface to build + test. The panel-swap reuses an existing, tested pattern. |
| **Direct download, no preview** | One-click, but the user can't see the generated module before it lands in Downloads; a read-only preview is cheap and reassuring. |
| **Expose `constName` / `typesImport` fields** | No current consumer needs a non-default shape; the mm-website modules expect exactly the defaults. YAGNI — the options remain available in code for a future consumer. |
| **Server-side export route** | Unnecessary — the serializer is pure and the deck is already in the browser store. A route would add I/O and a round-trip for zero benefit. |
